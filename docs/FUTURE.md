## F1 智能路由的有效性假设(原 H1,项目核心赌注)

**状态:未承诺 / 未排期 / 非当前范围**(注意:路由**机制**——规则引擎、SingleModelRouter 可替换性、决策面板——是核心规格并将实现;本条登记的是「智能路由能省成本保质量」这一**效果主张**)

- **想法描述**:在真实编码任务上,「中小模型 + 规则路由 + 本 harness」相对「单一大模型 + 本 harness」,可在质量保持 ≥90% 的前提下把成本降至 ≤50%。含「小模型起草 + 大模型精修」动态化的收益主张。这是项目差异化定位的存在理由(竞品空白的证据见 report_v2.md §5)。
- **为何当前不落地**:
  - 唯一支撑证据(LMSYS RouteLLM,2024-07:MT Bench 成本↓>85% 保留 95% GPT-4 质量,跨模型对零样本泛化)**全部来自对话/知识类 benchmark**(MT Bench/MMLU/GSM8K,GPT-4 Turbo/Mixtral 时代),不含多轮编码 agent 任务——域外证据。
  - 90%/50% 两个阈值是未校准拍板值。
  - 验证实验自身缺件:① benchmark 接入层(跑 Aider polyglot / SWE-bench Lite 需要 harness 对接,工作量不在 MVP 6-8 人周内);② N=30 任务的 pass@1 差异检验统计效力未分析;③ 实验预算未定。
  - 相邻论断警示(调研中 0-3 否决):「单模型路由可达 3.66x 节省」不成立,不得引用级联节省数字。
- **毕业标准(决策门协议)**:
  1. 补齐前置:benchmark 接入层建成;统计方案(样本量/检验方法)成文;实验预算获批。
  2. 基准集:≥30 个真实仓库编码任务(Aider polyglot 子集或 SWE-bench Lite 抽样 + 自选 5-10 个本仓库任务),固定种子与提示。
  3. 两臂对照:A = `SingleModelRouter`(单大模型);B = v0 规则路由。同一 harness、同一工具集。
  4. 指标:质量 = 验收命令通过率(pass@1);成本 = 实际 API $/任务;附延迟 p50。
  5. 判定:B 质量 ≥ 0.9 × A **且** B 成本 ≤ 0.5 × A → 毕业,提交 spec-reviewer;仅质量达标 → 调参重跑一轮;质量不达标 → 智能路由降级为可选功能,项目价值主张重估。
  6. 毕业前纪律:对外表述只称「路由机制可用」,不称「省成本保质量」。

## F2 学习型路由(RouteLLM 式训练管线)

**状态:未承诺 / 未排期 / 非当前范围**

- **想法描述**:用训练好的路由器模型替代 v0 规则表,按 query 难度在线选模(RouteLLM 证明该组件在对话域可学习且跨模型对泛化)。
- **为何当前不落地**:编码域的偏好/难度标注数据**不存在**(自采?公开 benchmark 改造?未定);训练与离线评估管线零设计;且其前提——规则路由不足以捕获收益——尚未被证明(原 H2)。缺的不是描述,是前提。
- **毕业标准**:
  1. F1 已毕业(否则路由本身价值未证,遑论学习型)。
  2. F1 实验中加跑 oracle 对照(每题事后选最便宜的能通过的模型):若规则路由收益 < oracle 收益的 60%,证明规则有显著遗留空间,学习型路由立项理由成立。
  3. 训练数据来源落实并完成最小数据集构建。
  4. 离线评估:学习型路由器在留出集上显著优于 v0 规则(指标与显著性标准立项时定)。
  5. 通过 spec-reviewer。

## F3 模型能力在线探针

**状态:未承诺 / 未排期 / 非当前范围**

- **想法描述**:用小样本任务实测模型的 code_gen 能力与 latency,自动回写 `ModelProfile`(`source = Probed`),替代/校准人工静态填写,解决画像保鲜问题。
- **为何当前不落地**:code_gen 探针的任务集、评分方法、样本量、成本上限全部未定义;**小样本实测"代码能力"的信度本身无支撑**(几个样本测不出能力),调研确认无成熟先例可抄。latency 探针(固定请求测 p50)是低风险子项,可先行毕业。
- **毕业标准**:
  1. 最小探针协议成文:任务集构成、评分函数、样本量、单次探针成本上限。
  2. 信度证据:同一模型重复探测的重测一致性数据(阈值立项时定);与公开 benchmark 排序的相关性检查。
  3. latency 子项可单独走简化流程:协议 + 一轮实测数据即可提交 spec-reviewer。
  4. 通过 spec-reviewer。

## F4 语义验证(LLM 评审员)

**状态:未承诺 / 未排期 / 非当前范围**

- **想法描述**:在机械验证(退出码判据)之上,引入 LLM 自评/评审员模型判断任务产出的语义正确性(如"改动是否符合意图"),作为模型路由层插件,不进内核。
- **为何当前不落地**:LLM-as-judge 在编码任务上的信度无数据;零规格、零验收标准;若与机械验证并列会污染 verify 的可测定义(已从核心规格的 verify 定义中移出)。
- **毕业标准**:
  1. 信度数据:在有机械判据的任务集上,评审员判定与机械验证结果的一致率及已知偏差表(一致率阈值立项时定)。
  2. 规格成文:评审员的输入契约、输出格式(可 schema 校验)、失败/分歧时的仲裁规则(机械验证恒为最终判据)。
  3. 成本影响评估:评审调用的额外开销进入预算机制。
  4. 通过 spec-reviewer。
